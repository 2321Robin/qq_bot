# QQ 群自定义机器人设计

## 目标

构建一个用于普通 QQ 群的自定义机器人。第一版聚焦三个能力：

- 群聊命令回复。
- 用户明确触发时的 AI 聊天回复。
- 向配置好的群发送定时消息。

机器人先在本地 Windows 跑通，项目结构保留以后迁移到服务器或 Docker 的空间。

## 技术栈

- QQ 客户端桥接：NapCatQQ。
- 通信协议：OneBot v11。
- 连接方式：反向 WebSocket，由 NapCatQQ 连接到 NoneBot2。
- 机器人框架：NoneBot2 + OneBot v11 适配器。
- 开发语言：Python。

这个组合适合普通 QQ 群机器人，支持文本、图片、艾特等常见消息段，并且能把自定义逻辑拆成独立 Python 插件。

## 架构

```text
QQ Group
  -> NapCatQQ
  -> OneBot v11 reverse WebSocket
  -> NoneBot2 app
  -> Bot plugins
```

机器人逻辑拆成小插件：

- `commands`：处理 `/help`、`/ping` 等简单命令和规则回复。
- `ai_chat`：把用户提示词发送给配置好的 AI 模型 API，并返回回复。
- `scheduler`：按固定时间向指定群发送消息。
- `config`：保存群号、管理员 QQ、AI API 配置和定时任务配置。

## 消息流程

命令回复流程：

```text
用户在群里发送 /ping
  -> NoneBot 收到群消息事件
  -> commands 插件匹配命令
  -> 机器人回复文本
```

AI 聊天流程：

```text
用户艾特机器人，或使用 AI 前缀发消息
  -> ai_chat 插件提取提示词
  -> 插件调用 AI API
  -> 机器人在群里回复
```

定时任务流程：

```text
到达配置好的时间
  -> scheduler 插件读取目标群列表
  -> 机器人向每个配置好的群发送消息
```

## 触发规则

AI 聊天默认不响应每一条群消息。为了避免刷屏，第一版只在以下条件之一满足时触发：

- 用户艾特机器人。
- 消息以配置前缀开头，例如 `ai `。

命令使用清晰前缀，例如 `/help` 和 `/ping`，避免和正常聊天冲突。

## 配置

第一版使用环境变量和一个简单项目配置文件。

需要的配置项：

- NoneBot2 监听地址、端口和 OneBot v11 反向 WebSocket 路径。
- NapCatQQ 侧反向 WebSocket 连接地址。
- `AI_API_KEY`：模型服务商 API Key。
- `AI_BASE_URL`：OpenAI 兼容服务商地址。
- `AI_MODEL`：使用的模型名称。
- 启用机器人的群号列表。
- 管理员 QQ 列表，为后续管理命令预留。
- 定时消息内容和发送时间。

API Key 等密钥不能提交到源码里。项目应提供 `.env.example` 说明需要哪些变量，但不包含真实密钥。

## 错误处理

- AI API 调用失败时，回复简短失败提示，不暴露原始异常。
- 群号不在启用列表时，忽略事件。
- 某个群定时发送失败时，继续发送其他群，并记录失败群号。
- 必需环境变量缺失时，启动阶段直接失败并输出清晰错误。

## 验证策略

第一版采用轻量验证：

- 本地启动 NoneBot2 应用。
- 用 NapCatQQ 通过 OneBot v11 反向 WebSocket 连接。
- 在测试群验证 `/ping` 能回复。
- 验证 AI 聊天只在艾特或前缀触发。
- 先用短间隔测试定时消息，再改成真实每日时间。

如果项目后续变大，再为命令解析和 AI 提示词提取添加单元测试。

## 第一版不做

- 禁言、踢人、拉黑、撤回等群管理能力。
- 复杂图片卡片生成。
- 持久化聊天记忆。
- Web 管理后台。
- 多账号机器人管理。

这些能力等基础机器人稳定运行后再加。

## 已确定决策

第一版固定使用 NapCatQQ、OneBot v11 反向 WebSocket、NoneBot2 和 Python。LLOneBot、QQ 官方机器人 API、图片卡片生成和群管理功能不进入第一版实现范围。
